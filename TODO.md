## Ferri app

- **Wire the Finder file-promise completion status icon.** `FilePromiseDragSourceView` (`Ferri/Ferri/Views/FilePromiseDragSource.swift`) fulfills promises but doesn't report incremental progress back to `NSFilePromiseProvider`/Finder, so Finder's progress/completion badge on the destination file never updates during a drag-drop download. Best tested against a large remote file, since small test files finish before the badge would visibly update.
- **Consider one aggregate transfer row per directory drag.** `downloadDirectoryRecursively` (`Ferri/Ferri/Views/FilePromiseDragSource.swift`) calls `TransferQueueViewModel.downloadAndWait` once per file, so dragging a deep tree to Finder floods the transfer pane with one row per file. A single row for the whole directory would read better, but the queue's `TransferItem.progress` needs `fileSize` up front and a plain `listDirectory` walk gives no tree total — it would need a recursive size pass (`listDirectoryRecursively` already exists on `FTPClientProtocol`) before the first byte is downloaded, which delays the drop, and `TransferQueueViewModel` currently models one row as exactly one `downloadFile` task (`runDownload`/`tasks[id]`), so pause/resume/cancel over a multi-file row is a real change, not a display tweak. Decide whether the noise is worth that; per-file rows were chosen as the simple option, not the clearly right one.
- **Expand Ferri app test coverage.** `Ferri/FerriTests/FerriTests.swift` already has 36 mock-based tests (`MockFTPClient`, no Docker/network needed — works fine under App Sandbox, contrary to the old note here). Audit for gaps: `FilePromiseDragSourceView`'s drag/download logic and `TransferQueueViewModel`'s pause/resume/cancel paths look under-tested relative to the other ViewModels.
- **Surface a clear error when a saved connection's initial directory is misconfigured.** `MainView` (`Ferri/Ferri/Views/MainView.swift:146`) calls `fileBrowserViewModel.loadDirectory(at: server.initialDirectoryPath ?? "")` right after connecting. If that path doesn't exist or isn't listable on the server, the user sees a raw, unhelpful `"The operation couldn't be completed. (FTPClient.SFTPClientError error 3.)"` alert — traced live to a misconfigured `initialDirectoryPath` on a connection to `server.ashelia.xyz`. Root cause is two-fold: (1) `SFTPClientError` (`FTPClient/Sources/FTPClient/SFTPClient.swift:14`) only conforms to `CustomStringConvertible`, not `LocalizedError`, so its `description` never reaches the alert shown to the user; and (2) there's no fallback when the configured initial directory fails — it should fall back to `/` (or the server's actual home directory) and surface a dismissible warning instead of leaving the browser on a hard error. Fix both: make `SFTPClientError` conform to `LocalizedError` (`errorDescription`), and add a fallback-to-root path in the post-connect load.
- **Add column resizing and column visibility toggle for the file browser.** `FileBrowserView` currently allows resizing only the Name column via `resizable()`; the other columns (date, size, type) have fixed widths set in `TableColumnWidth` definitions. Add the ability to: (1) resize all columns via drag handles; (2) choose which columns to show/hide via a preferences/contextual menu. This likely requires maintaining column width preferences in `FileBrowserViewModel` (or a new `FileBrowserPreferences` user-defaults store) and modifying the column definitions in `FileBrowserView` to bind their widths to state. See the Table modifier and related tools in SwiftUI documentation for `TableColumnWidth.flexible(minimum:maximum:)` or `.fixed()` dynamic options.
- **Configure logging so logs survive double-clicking the built app.** Nothing in the app or `FTPClient` package ever calls `LoggingSystem.bootstrap(...)`, so all `Logger(label:)` instances (`com.ftpclient.client`, `com.ftpclient.sftp`, `com.ftpclient.sftp.protocol`, `com.ferri.drag-source`) fall back to swift-log's default `StreamLogHandler.standardError` — plain stderr. That's fine when launching from Xcode or from Terminal, but stderr goes nowhere when the user double-clicks `Ferri.app` from Finder, so there's no way to retrieve logs after the fact for a failure report (e.g. debugging a failed SFTP connection). Bootstrap `LoggingSystem` early in `FerriApp` (or `FTPClient.shared`'s init) with a handler that also writes to macOS unified logging (`os.log`/`Logger` from `os`, viewable later in Console.app via `log show`/`log stream`) or to a file under Application Support, so logs are retrievable regardless of how the app was launched.
